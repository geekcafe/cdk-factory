# S3 CORS Presigned URL Fix — Bugfix Design

## Overview

The `S3BucketConstruct` in cdk-factory creates S3 buckets but has no support for CORS configuration. This means buckets that need browser-based presigned URL uploads fail because the browser's mandatory OPTIONS preflight request receives a 403 Forbidden. The fix adds a `cors_rules` property to `S3BucketConfig` (parsed from JSON config) and passes the resulting `s3.CorsRule` list to the `s3.Bucket` constructor in `S3BucketConstruct`. The change is additive and opt-in — buckets without `cors_rules` in their config remain unchanged.

## Glossary

- **Bug_Condition (C)**: A bucket is deployed via cdk-factory with `cors_rules` specified in its JSON config, but the construct ignores them — resulting in no CORS on the bucket
- **Property (P)**: When `cors_rules` are specified, the deployed bucket SHALL have matching `s3.CorsRule` entries so that browser preflight requests succeed
- **Preservation**: Buckets without `cors_rules` config, buckets using `use_existing`, and all other bucket properties (encryption, versioning, SSL, etc.) must remain completely unchanged
- **S3BucketConfig**: The configuration class in `cdk_factory/configurations/resources/s3.py` that parses bucket JSON config into typed properties
- **S3BucketConstruct**: The construct in `cdk_factory/constructs/s3_buckets/s3_bucket_construct.py` that creates or imports S3 buckets using `S3BucketConfig`
- **Presigned URL**: A time-limited URL generated by the backend that allows a browser to upload directly to S3 without backend credentials
- **CORS Preflight**: The browser's automatic OPTIONS request sent before cross-origin PUT/POST requests to verify the server allows the operation

## Bug Details

### Bug Condition

The bug manifests when a consumer specifies `cors_rules` in their S3 bucket JSON configuration and deploys via cdk-factory. The `S3BucketConfig` class has no `cors_rules` property, and `S3BucketConstruct` does not pass any `cors` parameter to `s3.Bucket()`. The bucket is created without CORS, causing all browser cross-origin requests to fail with 403 on the OPTIONS preflight.

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type S3BucketDeploymentConfig
  OUTPUT: boolean

  RETURN input.deployed_via_cdk_factory = true
         AND input.use_existing = false
         AND input.json_config.cors_rules IS NOT NULL
         AND input.json_config.cors_rules IS NOT EMPTY
         AND bucket_created_without_cors(input)
END FUNCTION
```

### Examples

- **Upload bucket with CORS config**: Config specifies `cors_rules: [{allowed_methods: ["GET","PUT","POST"], allowed_origins: ["*"], allowed_headers: ["*"]}]` → Bucket is created without CORS → Browser OPTIONS to `bucket.s3.amazonaws.com` returns 403 instead of 200
- **Multiple origins**: Config specifies `cors_rules: [{allowed_origins: ["https://app.example.com", "https://staging.example.com"]}]` → Bucket ignores these → Preflight fails from both origins
- **Exposed headers**: Config specifies `cors_rules: [{exposed_headers: ["ETag", "Date"]}]` → Bucket has no CORS → Browser cannot read ETag from upload response
- **No CORS config (not a bug)**: Config has no `cors_rules` key → Bucket created without CORS → This is correct behavior (no browser uploads needed)

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- Buckets without `cors_rules` in their JSON config must continue to be created without any CORS configuration
- Buckets with `use_existing: true` must continue to be imported without modification
- All existing bucket properties (encryption, versioning, removal_policy, block_public_access, enforce_ssl, access_control, public_read_access, auto_delete_objects, event_bridge) must continue to be applied identically
- The bucket naming, SSM parameter export, and resource type tagging must remain unchanged

**Scope:**
All S3 bucket configurations that do NOT include a `cors_rules` key should be completely unaffected by this fix. This includes:
- Buckets used only for backend-to-backend operations (no browser access)
- Static website hosting buckets (CORS handled by CloudFront)
- Log buckets
- Any bucket with `use_existing: true`

## Hypothesized Root Cause

Based on the bug description, the root cause is straightforward:

1. **Missing Config Property**: `S3BucketConfig` has no `cors_rules` property — the JSON config key is never read or parsed. Even if a consumer adds `cors_rules` to their JSON, it's silently ignored.

2. **Missing Construct Parameter**: `S3BucketConstruct` does not pass a `cors` parameter to `s3.Bucket()`. The CDK `s3.Bucket` constructor supports `cors: list[s3.CorsRule]` but it's never supplied.

3. **No Mapping Logic**: There is no code to convert a JSON cors rule definition (with string method names like `"GET"`, `"PUT"`) into CDK `s3.CorsRule` objects with `s3.HttpMethods` enum values.

4. **Pattern Gap**: The existing `lifecycle_rules` property in `S3BucketConfig` demonstrates the intended pattern (list of dicts from JSON) but is also not yet wired into the construct — CORS follows the same gap.

## Correctness Properties

Property 1: Bug Condition — CORS Rules Applied When Configured

_For any_ S3 bucket deployment where `cors_rules` is specified in the JSON config (isBugCondition returns true), the fixed `S3BucketConstruct` SHALL create the bucket with `s3.CorsRule` entries that match the configured allowed_methods, allowed_origins, allowed_headers, exposed_headers, and max_age values.

**Validates: Requirements 2.1, 2.2, 2.3**

Property 2: Preservation — Buckets Without CORS Config Unchanged

_For any_ S3 bucket deployment where `cors_rules` is NOT specified in the JSON config (isBugCondition returns false), the fixed `S3BucketConstruct` SHALL produce an identical bucket to the original construct — no CORS rules applied, and all other properties (encryption, versioning, removal_policy, block_public_access, enforce_ssl, access_control) remain the same.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4**

## Fix Implementation

### Changes Required

**File**: `cdk-factory/src/cdk_factory/configurations/resources/s3.py`

**Class**: `S3BucketConfig`

**Specific Changes**:
1. **Add `cors_rules` property**: Parse the `cors_rules` list from the JSON config dict. Each rule is a dict with keys: `allowed_methods` (list of strings), `allowed_origins` (list of strings), `allowed_headers` (list of strings, optional), `exposed_headers` (list of strings, optional), `max_age` (int, optional).

2. **Add HTTP method mapping**: Convert string method names (`"GET"`, `"PUT"`, `"POST"`, `"DELETE"`, `"HEAD"`) to `s3.HttpMethods` enum values.

3. **Return typed CorsRule list**: The property returns `list[s3.CorsRule]` ready for direct use in the construct, or an empty list if no CORS config is present.

---

**File**: `cdk-factory/src/cdk_factory/constructs/s3_buckets/s3_bucket_construct.py`

**Class**: `S3BucketConstruct`

**Specific Changes**:
4. **Pass `cors` parameter**: In the `s3.Bucket()` constructor call (new bucket path only), add `cors=self.bucket_config.cors_rules` if the list is non-empty, or omit/pass `None` if empty.

---

**File**: `Aplos-NCA-SaaS-IaC/cdk/configs/stacks/storage/s3-analysis-uploads.json`

**Specific Changes**:
5. **Add CORS configuration**: Add a `cors_rules` array to the `bucket` object matching the old deployment's configuration:
   ```json
   "cors_rules": [
     {
       "allowed_methods": ["GET", "POST", "PUT"],
       "allowed_origins": ["*"],
       "allowed_headers": ["*"],
       "exposed_headers": ["Date"],
       "max_age": 3600
     }
   ]
   ```

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, surface counterexamples that demonstrate the bug on unfixed code, then verify the fix works correctly and preserves existing behavior.

### Exploratory Bug Condition Checking

**Goal**: Surface counterexamples that demonstrate the bug BEFORE implementing the fix. Confirm that `S3BucketConfig` ignores `cors_rules` and `S3BucketConstruct` creates buckets without CORS even when config specifies it.

**Test Plan**: Write tests that instantiate `S3BucketConfig` with a `cors_rules` key and verify the config class has no way to access it. Synthesize a CDK stack with `S3BucketConstruct` using a config that includes `cors_rules` and verify the synthesized CloudFormation template has no `CorsConfiguration` property. Run these on the UNFIXED code to observe failures.

**Test Cases**:
1. **Config Parsing Test**: Create `S3BucketConfig({"name": "test", "cors_rules": [...]})` and attempt to access `cors_rules` property (will fail with AttributeError on unfixed code)
2. **Construct Synthesis Test**: Synthesize a stack with CORS config and check CloudFormation output for `CorsConfiguration` (will be absent on unfixed code)
3. **Method Mapping Test**: Verify string-to-enum mapping for HTTP methods (no mapping exists on unfixed code)

**Expected Counterexamples**:
- `S3BucketConfig` has no `cors_rules` attribute — accessing it raises `AttributeError`
- Synthesized CloudFormation template for a bucket with `cors_rules` in config contains no `CorsConfiguration` property
- Confirms root cause: config is silently ignored

### Fix Checking

**Goal**: Verify that for all inputs where the bug condition holds, the fixed construct produces a bucket with correct CORS rules.

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  config ← S3BucketConfig(input.json_config)
  ASSERT config.cors_rules IS NOT EMPTY
  ASSERT len(config.cors_rules) = len(input.json_config.cors_rules)
  FOR EACH rule IN config.cors_rules DO
    ASSERT rule.allowed_methods MATCHES input expected methods
    ASSERT rule.allowed_origins MATCHES input expected origins
  END FOR
  template ← synthesize(S3BucketConstruct(config))
  ASSERT template.bucket.CorsConfiguration.CorsRules IS NOT EMPTY
END FOR
```

### Preservation Checking

**Goal**: Verify that for all inputs where the bug condition does NOT hold, the fixed construct produces the same result as the original construct.

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT S3BucketConstruct_original(input) = S3BucketConstruct_fixed(input)
  // Specifically:
  // - No CorsConfiguration in synthesized template when cors_rules absent
  // - All other bucket properties identical in both versions
END FOR
```

**Testing Approach**: Property-based testing is recommended for preservation checking because:
- It generates many combinations of bucket configs (with/without various properties) automatically
- It catches edge cases where CORS addition might accidentally affect other properties
- It provides strong guarantees that non-CORS configs produce identical CloudFormation output

**Test Plan**: Synthesize buckets with the original and fixed constructs using configs WITHOUT `cors_rules`, then compare the CloudFormation templates to verify they are identical.

**Test Cases**:
1. **No CORS Preservation**: Verify a config without `cors_rules` produces a bucket with no `CorsConfiguration` in the template
2. **Empty CORS Preservation**: Verify a config with `cors_rules: []` produces a bucket with no `CorsConfiguration`
3. **Other Properties Preservation**: Verify encryption, versioning, removal_policy, block_public_access, enforce_ssl are identical with and without the fix
4. **Use Existing Preservation**: Verify `use_existing: true` buckets are imported without any CORS modification attempt

### Unit Tests

- Test `S3BucketConfig.cors_rules` returns correct `s3.CorsRule` list for valid config
- Test `S3BucketConfig.cors_rules` returns empty list when key is absent
- Test `S3BucketConfig.cors_rules` returns empty list when key is `[]`
- Test HTTP method string-to-enum mapping for all valid methods (GET, PUT, POST, DELETE, HEAD)
- Test invalid HTTP method string raises or is skipped gracefully
- Test `max_age` defaults to None when not specified
- Test multiple CORS rules in a single config

### Property-Based Tests

- Generate random valid CORS configs (random subsets of methods, random origin strings, random header lists) and verify `S3BucketConfig.cors_rules` produces matching `s3.CorsRule` objects
- Generate random bucket configs WITHOUT `cors_rules` and verify synthesized templates are identical between original and fixed constructs
- Generate random bucket configs WITH `cors_rules` and verify synthesized templates contain correct `CorsConfiguration`

### Integration Tests

- Deploy a bucket with CORS config via cdk-factory and verify OPTIONS preflight returns 200 with correct headers
- Deploy a bucket without CORS config and verify no CORS headers are returned
- End-to-end: generate presigned URL, perform browser-style upload (OPTIONS then PUT) against CORS-enabled bucket
